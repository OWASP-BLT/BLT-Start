Local Setup

1

Fork & clone the repository

git clone https://github.com/<YOUR-USERNAME>/BLT.git
cd BLT
2

Create a virtual environment & install dependencies

python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt
3

Configure environment & run migrations

cp .env.example .env
# Edit .env with your local values
python manage.py migrate
python manage.py runserver

Full instructions are in the project README .

Good First Issues

Good first issues are deliberately small, well-scoped tasks. They let you learn the codebase, the PR workflow, and the community norms without being overwhelming.

Before you pick an issue

  • Read the issue description and all comments — someone may have already started work.
  • Check if the issue is already assigned; if so, comment to ask whether help is welcome.
  • Ask clarifying questions in the issue before coding — this saves round-trips in review.
  • Reproduce the bug (or understand the feature request) locally first.

Creating a Pull Request

1

Create a feature branch

Never commit directly to main.

git checkout -b fix/your-descriptive-branch-name
2

Write focused, small commits

Each commit should do one thing. Use the conventional commits style: 

git commit -m "fix: correct typo in login error message"
git commit -m "feat: add dark-mode toggle to navbar"
3

Sync your fork before pushing

git fetch upstream
git rebase upstream/main
4

Open the Pull Request

  • Link the issue with Closes #<issue-number> in the PR description.
  • Add screenshots or a short screen recording for UI changes.
  • Fill in the PR template completely — skip no sections.
  • Pass all CI checks before requesting a review.
5

Respond to review feedback

Address every comment, mark resolved conversations, and push updates to the same branch. Do not open a new PR for review fixes.

Code Review Expectations

As a Contributor

  • Keep PRs small and focused on one concern.
  • Write or update tests for every change.
  • Self-review the diff before requesting a reviewer.
  • Be patient — reviewers are volunteers.

As a Reviewer

  • Leave kind, specific, actionable feedback.
  • Distinguish blocking issues from suggestions.
  • Approve promptly when changes are satisfactory.
  • Mentor first-timers — celebrate their wins.

Style Guide — Buttons

Two button variants must be used consistently across all BLT UI surfaces.

Markup

<!-- Primary -->
<button class="px-4 py-2 rounded-md bg-[#E10101] text-white text-sm font-medium
               hover:bg-red-700 focus:ring-2 focus:ring-offset-2 focus:ring-red-600">
  Primary Button
</button>

<!-- Outline -->
<button class="px-4 py-2 rounded-md border border-[#E10101] text-[#E10101] text-sm font-medium
               hover:bg-[#E10101] hover:text-white focus:ring-2 focus:ring-offset-2 focus:ring-red-600">
  Outline Button
</button>

Style Guide — Icons

BLT uses Font Awesome 6. Apply aria-hidden="true" to decorative icons. For standalone icons (buttons without text), provide an aria-label on the button.

fa-bug
fa-code-pull-request
fa-brands fa-github
fa-star
fa-shield-halved
fa-lock
fa-circle-check
fa-triangle-exclamation

Accessibility usage

<!-- Decorative: hidden from assistive tech -->
<i class="fa-solid fa-bug" aria-hidden="true"></i>

<!-- Icon button: label on the button wrapper -->
<button aria-label="Report a bug">
  <i class="fa-solid fa-bug" aria-hidden="true"></i>
</button>

<!-- Icon + visible text: icon is decorative -->
<button>
  <i class="fa-solid fa-bug" aria-hidden="true"></i>
  Report Bug
</button>

For Mentors

Mentors are vital to keeping BLT's contributor pipeline healthy. Here is how to set up new contributors for success:

Label issues well

Apply good first issue and add a short comment describing the expected outcome and files likely to change.

Set scope in comments

Paste a short comment with: files to edit, acceptance criteria, and estimated effort. Saves back-and-forth.

Respond within 48 hours

Slow reviews discourage newcomers. Aim for at least an acknowledgement within two business days.

Celebrate merged PRs

A short "great work!" comment or Slack message goes a long way in retaining contributors.
